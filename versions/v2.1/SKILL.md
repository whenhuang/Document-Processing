# 公文格式排版

## ⚠️ 核心原则（不可违反）

**源文档只读原则**：任何时候不得以任何方式修改源文档。包括但不限于：
- 不得将排版结果 `save()` 回源文档路径
- 不得创建外部脚本直接覆写源文档（如 `doc.save(原路径)`）
- 层级编号修正、引号规范化、项目符号替换等所有内容级改动，只作用于排版输出文件

输出文件默认生成在源文档所在目录，不与源文件混同。

## 概述

将未排版或格式混乱的 .docx 文档，按《党政机关公文格式》(GB/T 9704) 标准自动排版。脚本自动识别文档结构（标题、主送、层级标题、正文、附件、落款等），并逐段应用对应的字体、字号、行距、缩进。

## 触发场景

- 用户说"把这份文档按公文格式排版"
- 用户说"帮我格式化这篇公文"
- 用户提供一份没有格式的 .docx，要求按公文标准排版
- 用户说"按 GB/T 9704 排版"

## 工作流程

### Step 1: 确认输入文件

从用户消息中确认待排版的 .docx 文件路径。如果用户只给了文件名没有完整路径，用 Glob 在当前工作目录搜索。

> **路径兼容提示**：Windows 目录名常含中文全角引号（如 `1."智海云帆"系列`），Bash 无法直接传递。遇到此类路径，使用 Python `os.listdir()` + `subprocess.run()` 构造命令执行，避免转义问题。

### Step 2: 预览文档结构（推荐）

先使用 `--preview` 参数预览文档结构，检查层级是否正确：

```bash
python scripts/format_gongwen.py <输入文件.docx> --preview
```

预览输出包含：
- 所有段落的类型识别结果（标题、层级标题、正文等）
- **层级问题提示**：如"一、"下直接出现"1、"而未经过"（一）"，会标出建议调整的位置
- **非标项目符号**：如 `●` `·` `◆` 开头的段落，会被标记为疑似小标题

预览后按以下规则处理诊断项：

| 诊断项 | 处理方式 | 说明 |
|--------|----------|------|
| 标题断行 | **自动** | 自动选择语义断点（动词组合前），且过滤掉会割裂固定名词（如"国家发改委"）的候选；不询问 |
| 层级标题拆分 | **需确认** | 标题+正文混合段落是否拆分，默认全部拆分 |
| 层级跳跃修正 | **需确认** | "一、"下跳级出现"1、"是否修正，默认全部修正 |
| 引号规范化 | **自动** | 半角引号自动替换为全角配对引号，不询问 |
| 空段落删除 | **自动** | 段间空行自动删除，不询问 |
| 非标符号（◆●·）开头段落 | **需确认** | 检测到后随"层级标题拆分"一起呈现，询问是否转为正式标题 |
| 行距/页边距/页码 | **自动** | 排版时自动修正，不询问 |

**仅对「层级标题拆分」「层级跳跃修正」「非标符号」三项发起确认。** 确认完毕后直接进入排版，不再追加其他询问。

**确认规范（重要）**：所有 AskUserQuestion 必须标注推荐选项，在 label 后追加 ` (Recommended)`。推荐逻辑：

| 确认项 | 推荐选项 | 理由 |
|--------|----------|------|
| 层级跳跃修正 | "全部修正" | 符合公文层级规范，标准行为 |
| 层级标题拆分 | "全部拆分" | 标题+正文混合段落拆开更清晰 |
| 非标符号（◆●·等）转标题 | "转为二级标题" | 语义上通常是小标题，转正式编号更规范 |

如果层级有问题（如"一、"下直接出现"1、"），请在正式排版时使用 `--fix-hierarchy` 参数，脚本会在输出文件上自动修正层级编号。**绝不修改源文档。**

### Step 3: 执行排版

确认结构无误后，去掉 `--preview` 执行正式排版。如有层级跳跃问题，附带 `--fix-hierarchy`：

```bash
# 标准排版（输出到源文档同目录，自动加 _已排版 后缀）
python scripts/format_gongwen.py <输入文件.docx>

# 排版 + 层级编号修正
python scripts/format_gongwen.py <输入文件.docx> --fix-hierarchy

# 如需指定输出路径，必须放在源文档同目录
python scripts/format_gongwen.py <输入文件.docx> <同目录/输出文件.docx>
```

- 第一个参数：必须，待排版的 .docx 文件路径
- 第二个参数：可选，输出路径。**省略时自动生成在源文档同目录**（`<源文件名>_已排版.docx`）；如需指定，必须放在源文档所在目录
- `--fix-hierarchy`：自动修正"一、"下直接出现"1、"等层级跳跃问题（仅作用于输出文件）

注意：使用工作空间中的 Python 环境执行，确保已安装 `python-docx`。如未安装，先执行 `pip install python-docx`。

### Step 3.5: 后处理非标符号（仅当用户确认转换时）

当用户确认将 `◆` `●` `·` 等非标符号转为正式标题后，脚本排版不会自动处理这些段落。需对输出文件用 python-docx 做后处理：

1. 打开 `_已排版.docx` 输出文件
2. 遍历段落，匹配以非标符号开头的段落
3. 替换符号为 `（一）（二）...` 编号，应用二级标题格式（楷体_GB2312、三号、加粗、两端对齐、首行缩进）
4. 保存

```python
# 后处理示例：◆ → （一）（二）
from docx import Document
from docx.shared import Pt
from docx.enum.text import WD_ALIGN_PARAGRAPH
from docx.oxml.ns import qn

doc = Document('输出文件_已排版.docx')
labels = ['（一）', '（二）']  # 按需扩展
counter = 0
for p in doc.paragraphs:
    if p.text.strip().startswith('◆'):
        new_text = labels[counter] + p.text.strip().lstrip('◆　 \t')
        for run in p.runs:
            run.text = ''
        p.runs[0].text = new_text if p.runs else p.add_run(new_text)
        for run in p.runs:
            run.font.name = '楷体_GB2312'
            run._element.rPr.rFonts.set(qn('w:eastAsia'), '楷体_GB2312')
            run.font.size = Pt(16)
            run.bold = True
        p.alignment = WD_ALIGN_PARAGRAPH.JUSTIFY
        pf = p.paragraph_format
        pf.first_line_indent = Pt(32)
        pf.line_spacing = Pt(28)
        counter += 1
doc.save('输出文件_已排版.docx')
```

> **注意**：只匹配指定符号（如 `\u25c6` = ◆），不要匹配 `（` 开头段落，避免误伤已编号的标题。

### Step 4: 呈现结果

排版完成后，执行以下两项：

**① 告知输出路径（不弹保存卡片）**：
排版结果自动保存在源文档同目录（`<源文件名>_已排版.docx`）。仅在回复中告知路径，**不调用 `present_files`**（用户不需要弹窗选择保存位置）。

**② 输出修改摘要**：
提供结构化摘要，分类列示本次修改内容：
- 识别出的文档结构（标题、几级标题、是否有附件等）
- **排版修改摘要**：格式修正/内容规范化/层级修正/未处理提示，使用户无需打开文件即可了解全部改动

## 排版修改摘要（v2 新增）

排版完成后自动输出一份修改摘要，分为四类：

1. **格式修正**：页边距/页码/行距/段前段后/表格等全局格式变更
2. **内容规范化**：引号替换处数、标题断行位置等文本级修改
3. **层级修正**：标题段落拆分记录、层级编号修正（--fix-hierarchy）的每一处映射
4. **未处理提示**：bullet_heading 等保留为正文的段落，提醒用户关注

摘要格式示例：


## 排版规范参考

脚本按以下标准排版。如需微调，参考此表手动修改。

**对齐总则（重要）**：
- 文章标题：居中对齐（CENTER）
- 正文部分（普通正文、二~四级标题、结尾用语）：**两端对齐（JUSTIFY）**——这是用户的硬性排版要求，任何改动都不得破坏
- 一级标题（一、二、）：**左对齐（LEFT）**（v2.1 修正，由 JUSTIFY 改 LEFT）
- 公文格式要素（主送机关、附件说明、联系人、版记）：左对齐（LEFT），保持不变
- 落款单位/落款日期：右对齐；发文字号：居中；签发人：右对齐；表格单元格：左对齐

| 场景 | left_indent | first_line_indent | 视觉效果 |
|------|-------------|------------------|---------|
| 1个附件 | 0 | 2字符 | 第二行从首行首字起，与"附件"二字对齐 |

页边距：上 3.7cm / 下 3.5cm / 左 2.8cm / 右 2.6cm（A4纸）。

**段前段后**：均为 0（避免 Word 默认段距造成的视觉空隙）。

**页码**：宋体 四号(14pt)，格式 "— 1 —"（两边短横、中间数字），奇数页右下角，偶数页左下角（页脚外侧）。

## 自动识别规则

脚本按以下优先级识别段落类型：

1. **标题**：文档首段，或以"关于...的报告/请示/函"等文种关键词开头
2. **层级标题**：正则匹配一、、（一）、数字.、（数字）等模式（**优先于标题关键词**，避免正文中"关于...的指导意见"等被误判）
3. **主送机关**：文档前部、长度<25 字符、以"："结尾
4. **附件**：以"附件"开头的段落
5. **落款日期**：文档末尾的"XXXX年XX月XX日"格式
6. **落款单位**：日期上方的非标点文本
7. **联系人**：包含"联系人："的段落
8. **正文**：其余非空段落
9. **表格**：遍历文档所有表格，统一为仿宋小四、表头加粗、细单线边框（六向）、单元格左对齐，列宽与已有单元格对齐方式保留
10. **发文字号**：含 `〔YYYY〕…号` 的段落（如 `深港集团发〔2026〕5号`）
11. **签发人**：含"签发人"的段落（上行文）
12. **版记**：以"抄送/印发"开头的段落（抄送机关、印发机关及日期）

## 标题段落自动拆分

如果某个标题段落包含标题+正文（如"1、私有化企业微信：员工日常工作的入口..."），脚本会**自动在第一个有效拆分点处断开**为两段。拆分点优先级：`：` > `——` > `。`。

- 拆分点必须保证：标题部分 ≥ 4 字符 + 正文部分 ≥ 15 字符（`MIN_BODY_LENGTH` 阈值）
- 正文不足 15 字时视为标题后缀/副标题，不拆分。例：`一、"海规百问"——制度咨询智能体` 中 `——制度咨询智能体` 仅 7 字，保留在标题内
- 原段落保留为标题（如"1、私有化企业微信"）
- 在其后插入新段落承载正文

适用于 heading1（一、）、heading2（（一））、heading3（1.）等所有层级。

## 标题断行智能保护

标题断行时，`_is_break_safe()` 函数会检查候选断点是否落在固定名词内部，自动排除不安全的候选。

**保护词库**（`UNBREAKABLE_PHRASES`，30+ 词）：

| 类别 | 示例 |
|------|------|
| 机构/地名 | `国家发改委` `深圳市` `广东省` `人民政府` `国务院` |
| 企业/品牌 | `深圳港` `集团` |
| 政策术语 | `高质量` `现代化` `信息化` `数字化` `社会主义` |
| 行业术语 | `海洋产业` `海洋强国` `研究成果` `交通运输` `科技创新` |

**工作原理**：对每个候选断点，提取断点前后 3-4 字窗口，逐条比对词库。若断点恰好落在某保护词的中间（如 `国家发`←断→`改委`），则跳过该候选。

**动词匹配扩展**：`TITLE_BREAK_POINTS` 正则已覆盖 `携打造推动构建助力共同联合开启深驱获荣斩夺取摘` × `选定入筑赋提深驱融聚造动力能升国市研创发展` 等双字组合，语义断点检测更灵敏。

## 新增预处理功能

排版前自动执行以下规范化处理：

| 处理项 | 说明 |
|--------|------|
| 引号规范化 | 半角 `""` `''` → 全角 `""` `''`（状态机：奇数次=开，偶数次=闭，正确区分左右引号） |
| 前后空白清理 | 清理所有段落 runs 的首尾空白字符（含 `\t` 制表符） |
| 空段落删除 | 删除文档 body 中所有纯空白 `<w:p>` 段落，消除段间空行；修改摘要中报告删除数量 |
| 长标题自动断行 | 标题超 18 字时，优先语义断点（动词组合前），再经安全过滤器排除割裂固定名词的候选；自动处理，不询问 |
| 项目符号检测 | 识别 `●` `·` `◆` `✦` 等非标符号开头的段落，在预览中标注，随确认流程询问是否转为正式标题 |
| **序号点号全角化（v2.1 新增）** | 标题（heading3/4）和附件（attachment）段落中的半角序号点 `1.` `2.` 自动转全角 `1．` `2．`；正文段落不处理 |
| **附件文本规范化（v2.1 新增）** | 附件段落自动去除首尾书名号 `《》`、去除文件名末尾中英文标点（。，．,;；） |
| **附件悬挂缩进分发（v2.1 新增）** | 按附件序号应用不同悬挂缩进：1个附件→无悬挂，附件1→4.5字符悬挂，附件2+→1.5字符悬挂 |

## 层级编号检测与修正

脚本在排版前自动检测层级跳跃问题（如"一、"下直接出现"1、"而未经过"（一）"），并在结构预览中标注建议。使用 `--fix-hierarchy` 参数可在排版时自动修正——将跳跃的 `1、2、...` 替换为 `（一）（二）...`，**修正仅作用于输出文件，不修改源文档**。

## CLI 参数

| 参数 | 作用 |
|------|------|
| `--preview` | 仅打印结构预览和诊断报告，不生成排版文件 |
| `--title-break=<pos>` | 指定标题断行位置（字符索引），如 `--title-break=18` |
| `--fix-hierarchy` | 自动修正层级跳跃（"一、"下直接出现"1、"等），在输出文件上将 `1、→（一）` 格式修正，**不修改源文档** |
| `--auto` | 自动模式：跳过所有确认环节，使用默认行为（适合 CLI 批处理） |

## 注意事项

- **源文档保护**：所有操作只读源文档、只写输出文件。永远不要用 `doc.save(源路径)` 或外部脚本覆写源文档。层级编号修正应使用 `--fix-hierarchy` 参数在输出文件上完成。
- 脚本依赖 `python-docx` 库，首次使用需安装
- 仅支持 .docx (OOXML) 格式；旧版 .doc 格式会被自动检测并提示转换
- 标题和落款的识别依赖内容特征，结构特殊的文档可能需要手动微调
- 多段落标题：首段为标题时，紧随其后的短续行（未以句号结尾、全文尚未出现一级标题）会自动并入标题，避免长标题被手工换行后误判为正文
- 三级标题拆分：`1.标签：内容` 形式的列表项会被拆为三级标题+正文两段（三级标题默认不加粗）；如需保留原单行加粗格式需手动调整
- 表格内的段落不进入正文分类，避免"序号"等被误判为层级标题或覆盖表格字体
- 空段落（段间空行）在排版预处理阶段自动删除，不会保留在输出文件中
- 对齐约定（不可偏离）：仅正文部分（普通正文 + 一~四级标题 + 结尾）两端对齐；主送机关、附件说明、联系人、版记保持左对齐；文章标题居中；落款单位/日期右对齐、发文字号居中、签发人右对齐、表格单元格左对齐。**切勿将主送机关/附件说明/联系人/版记误改为两端对齐**
- 方正小标宋简体、仿宋_GB2312、楷体_GB2312、黑体等字体需系统已安装
- 页码需在 Word 中按 F9 或保存后刷新才能正确显示
